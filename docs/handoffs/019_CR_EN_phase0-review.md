## 正確性問題

### P1 — SBER公式量測的是錯誤對象

`_sentence_break_error_rate`（`phase0_spike.py` L138–164 / `phase0_chunk_test.py` L212–228）：

計算「baseline sentences absent from hypothesis」，但這衡量的是*內容遺漏*，而非*句子邊界被截斷*。一個完整句子在chunk邊界處被切成兩半，兩半都進入不同chunk，set lookup不會捕捉此錯誤——hypothesis中無任何一個完整句子，因此SBER為1.0；但若chunk切割恰好不破壞句子完整性，SBER為0.0。兩者語義完全不同。

同時，set比較忽略重複出現句子（`full_set = set(full_sents)`），且不做文本正規化（大小寫、標點殘留），致使細微差異被計作broken。

建議：改用word-level edit distance近似，或至少在兩腳本間統一做`lower()` + strip標點後再比較。`phase0_chunk_test.py`的版本較優（有`lower()`），`phase0_spike.py`無此處理。

兩腳本應共用同一實現，現有重複代碼。

### P1 — Test 2 RTF目標單位錯誤

`phase0_spike.py` L325：
```python
target_durations = [5, 15, 30, 60, 120]  # minutes
```
但L334：
```python
target_sec = target * 60
```

docstring寫「RTF for various audio durations」，但若測試文件名為`5min.wav`、`60min.wav`（由`prepare_test_audio.py`生成），則duration_to_file匹配會找5分鐘 = 300秒的文件。匹配邏輯：`abs(d - target_sec) < 30`，即300秒±30秒。這是正確的。

但fallback路徑（L338–350）對前5個任意文件計算RTF，並用`pass: rtf <= 2.0`判定——若文件極短（LibriSpeech utterance約5–10秒），RTF本身沒有意義，`pass`字段會誤導。

建議：fallback路徑僅記錄RTF，去掉`pass`字段，或標記`"pass": null`。

### P1 — Test 5記憶體監控基線時機錯誤

`phase0_spike.py` L500：
```python
baseline_mb = _measure_rss_mb()
```

此行在`transcribe_mod.load_model()`**之後**執行（load_model在main() L773）——模型已載入記憶體，baseline已含模型佔用。因此`delta_mb`衡量的是「transcription run本身的增量」，而非「模型+運行總計對系統的影響」。

spec寫「peak RAM < model_baseline_mb + 500 MB」，`model_baseline_mb`語義不明——若指OS無模型時的基線，現有實現永遠測不到。若指「模型已載入後的基線+500MB」，則命名應修正且需在report中明確說明。

建議：在`load_model`前後各採樣一次，分別記錄`pre_load_rss_mb`、`post_load_rss_mb`、`peak_during_transcription_mb`，令PM可看到完整圖像。

### P2 — Test 6/7臨時文件洩漏

`phase0_spike.py` Test 6, L593：
```python
with tempfile.NamedTemporaryFile(suffix=".wav", delete=False) as tmp:
    tmp_wav = tmp.name
```
後有`os.unlink(tmp_wav)`，但若ffmpeg呼叫成功後soundfile.read拋出異常，`os.unlink`不執行。Test 7同樣問題（L705）。

建議：用`try/finally`或`contextlib.ExitStack`包裹。

## 可靠性問題

### P1 — `prepare_test_audio.py`的AISHELL下載提示不足

L173：
```python
log.info(f"AISHELL archive is large (~15GB). Download may take a while.")
```
15GB存入`output_dir`下，但腳本不檢查可用磁盤空間，也不在下載前確認。EN文件中提示「AISHELL約15GB，若磁盤不足可加`--no-aishell`」，但腳本本身無任何guard——下載失敗會被`except Exception as e: log.error(...)`靜默吞掉，令用戶困惑。

建議：下載前用`shutil.disk_usage`檢查可用空間，低於20GB時列印明確警告並提示`--no-aishell`。

### P1 — `generate_duration_files`的pool重複平鋪產生重複音頻

`prepare_test_audio.py` L311：
```python
repeats = int(target_sec / pool_sec) + 2
tiled = np.tile(pool, repeats)
```
120分鐘文件若source pool不足，整段音頻重複多次。對Test 2（RTF benchmark）影響不大，但對Test 1（chunk strategy）和Test 3（WER），重複片段使baseline text與chunked text的比較人工簡化，SBER可能偏低——所有策略都顯得「更好」。

建議：在report和README中加警告：「duration files generated by tiling — WER/SBER metrics may be optimistic」。若source material不足，應在manifest.json中標記。

### P2 — `_load_transcribe_module`重複載入的副作用

`phase0_spike.py` L80–91 / `phase0_chunk_test.py` L39–48：
`importlib.util.spec_from_file_location`每次呼叫都重新執行`transcribe.py`頂層代碼，包括：
- `logging.basicConfig(filename=log_path, ...)` — 覆寫已設定的logging配置
- `torch.import` 和 `os.environ.setdefault("KMP_DUPLICATE_LIB_OK", "TRUE")`

`phase0_chunk_test.py`在main()重新呼叫`_load_transcribe_module()`一次，影響尚小。但若未來測試腳本多次呼叫，logging會被靜默重定向至transcribe.log，phase0腳本的stderr輸出消失，調試困難。

建議：`_load_transcribe_module`加module cache：
```python
_transcribe_mod = None
def _load_transcribe_module():
    global _transcribe_mod
    if _transcribe_mod is not None:
        return _transcribe_mod
    ...
    _transcribe_mod = mod
    return mod
```

### P2 — `requirements-dev.txt`缺少`soundfile`和`numpy`

`phase0_spike.py`、`phase0_chunk_test.py`、`prepare_test_audio.py`均依賴`soundfile`和`numpy`，但`requirements-dev.txt`未列出。`phase0_chunk_test.py` docstring提及「Dependencies: soundfile, numpy (required)」，但安裝文件未跟進。

在CI乾淨環境中`pip install -r requirements-dev.txt`後直接運行會報`ImportError`。

建議：添加：
```
soundfile>=0.12.0
numpy>=1.24.0
```

## 與現有代碼一致性

### P2 — `_load_transcribe_module`的logging衝突（承上P2項）

`transcribe.py` L34–39設定`filename=log_path`的文件logging；`phase0_spike.py` L36–41設定console logging。兩者皆呼叫`logging.basicConfig`。Python的`basicConfig`第一次呼叫生效，後續無效——但`exec_module`執行`transcribe.py`頂層時若logging尚未初始化，會設定文件handler，此後phase0的console handler不生效。

實際影響取決於import順序，且難以重現。建議spike腳本改用：
```python
logging.basicConfig(...)
# 在_load_transcribe_module內，使用logging.getLogger()而非basicConfig
```
或在`exec_module`前臨時monkey-patch `logging.basicConfig = lambda **kw: None`。

### P1 — `transcribe.py`不暴露`load_model`返回模型物件

`transcribe.py`用全局變量`backend`/`encoder_sess`/`hf_model`管理狀態——`load_model()`返回`{"status": "ok"}`，side-effect寫入module全局。

`phase0_spike.py` L773正確呼叫`transcribe_mod.load_model(args.model_path)`，但因為importlib動態載入，模組全局在此module namespace中（非`sys.modules`），呼叫`transcribe_mod.transcribe()`時讀取的也是同一namespace的全局——**這是正確的**，無bug。

但若任何地方`import transcribe`（標準import），會創建獨立的module object，全局不共享，導致「model not loaded」錯誤。建議在`_load_transcribe_module`的docstring中明確寫：「Must use importlib path — do not import via sys.path」。

## Report格式

### P2 — `single_speaker_wer`的`status`字段語義模糊

`phase0_spike.py` L414–415：
```python
"status": "pass" if (results_en["has_data"] or results_zh["has_data"]) else "skip",
```
Test 3無hard threshold，status永遠是pass或skip，令summary中的`passed`計數虛高。PM看到「6 pass」時，Test 3的pass實為「有數據」而非「達標」。

建議：status改為`"done"`（有數據）或`"skip"`（無數據），不用`"pass"`——或在report加`"note": "observational — no pass threshold"`。

## Swift測試

### P1 — Test 8的round-trip測試缺少cleanup guard

`Phase0SpikeTests.swift` L56–78，`test_activationPolicy_roundTrip_remainsCoherent`：
若`XCTAssertEqual`失敗，`app.setActivationPolicy(originalPolicy)`（cleanup）不執行——因Swift測試失敗後繼續執行但XCTestCase不會自動恢復。這會影響同suite後續測試。

`test_activationPolicy_switchToRegular_succeeds`（L19–35）有正確的cleanup，round-trip版本應同等處理。

建議：用`defer`：
```swift
let originalPolicy = app.activationPolicy()
defer { app.setActivationPolicy(originalPolicy) }
```

### P1 — `test_activationPolicy_switchToAccessory_succeeds`未cleanup

`Phase0SpikeTests.swift` L39–51：測試結束時policy為`.accessory`，未恢復originalPolicy。若與其他會顯示Dock icon的測試同session運行，可能有視覺副作用（Dock icon消失/出現競態）。

建議：同上，加`defer { app.setActivationPolicy(originalPolicy) }`。

### P2 — App Nap測試的`performActivity`是同步API，XCTestExpectation多餘

`Phase0SpikeTests.swift` L103–114：`NSProcessInfo.performActivity(_:reason:using:)`在**當前thread同步執行block**（非async），因此`expectation.fulfill()`在`wait`之前已必然執行。`expectation`和`wait(timeout:5.0)`在此無實際意義，只增加閱讀複雜度。

無需修復（不影響正確性），但值得加comment說明此為同步call，或刪除expectation改為：
```swift
XCTAssertTrue(blockExecuted)
```

### P2 — `test_activationPolicy_switchToProhibited_doesNotCrash`未斷言任何行為

`Phase0SpikeTests.swift` L82–92：整個test body只是呼叫API，無任何XCTAssert（除implicit crash check）。若測試意圖是「不crash即pass」，應加comment說明；否則未來EN可能誤以為需補斷言。

代碼結構清晰，fallback路徑完整，與`transcribe.py`的接駁方式正確（importlib動態載入），JSON report格式符合spec需求。主要弱點集中在：

1. **SBER定義** — 量測到的不是chunk邊界截斷，而是內容遺漏，且兩腳本實現不一致（P1）
